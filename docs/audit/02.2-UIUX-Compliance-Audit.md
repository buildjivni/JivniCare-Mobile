# JivniCare Mobile — Engineering Audit, Phase 2.2: UI/UX Compliance Audit

**Audit type:** Read-only comparison of implemented screens/components against
`docs/07-Mobile-UX-Spec.md`, `docs/08-Design-System.md`, `docs/09-Component-Library.md`, and
`docs/10-UX-Writing-Guide.md`.
**Scope:** `audit-prep` branch, commit `df4085f11f8957d8cab72b98282360d0a3858b36`
**No code was modified as part of this audit.**

---

## How to read this report

Only **3 of the 16 documented screens** exist in code (Home, Doctor Profile, Booking — rough,
simplified subsets of S05, S07, and S08 respectively). Compliance percentages below are scored
against the _full documented spec_ for each screen, not against a reduced "MVP scope" — this
gives an honest picture of distance-to-spec, not just "does what exists work." The remaining 13
screens (S01–S04, S06, S09–S16) are scored 0% / Not Implemented and are listed in a single
summary table rather than repeated per-topic sections, since there is no code to evaluate against
each topic for them.

| Headline + doctor name                                                                                | ✅ Present                                                                                                                                        |
| Doctor summary card (avatar + name + specialty + clinic)                                              | ❌ Missing — only a plain text line with the doctor's name                                                                                        |
| Token preview (railway-style, progress bar, "X of Y slots", "N ahead")                                | ❌ **Entirely missing** — replaced by a **time-slot picker**, a fundamentally different booking model not present anywhere in the documented spec |
| Fee breakdown (Consultation Fee / Platform Fee struck-through FREE / Total Payable / savings message) | ❌ Missing                                                                                                                                        |

| 13 named blocked-state messages                                                                       | ✅ **Fully implemented** — the strongest-compliance area of the whole app (see below)                                                             |
| Loading ("Booking..." + spinner)                                                                      | ✅ Present, matches spec text exactly                                                                                                             |
| Success → navigate to S09 Token Tracking                                                              | ❌ Missing — success just shows an inline banner; there is no S09 screen to navigate to yet                                                       |

**Missing Items**

- The entire token-preview/fee-breakdown/consent-checkbox section of S08 — none of it exists.
- Navigation to S09 on success (S09 itself doesn't exist yet, so this is a downstream gap too).
- A `[Join Waitlist]` CTA on the `QUEUE_FULL`/`DAILY_LIMIT_REACHED` (doctor-level) error states.

**Incorrect Items**

- **This is the single largest spec deviation found in the entire audit.** `BookingWidget`
  implements a **time-slot selection UI** ("9:00 AM", "9:30 AM", etc.) — a booking model that does
  not exist anywhere in `05-Business-Rules.md` or `11-API-Contract.md`. The real JivniCare model
  (confirmed across both documents) is a **sequential queue-token system**: a patient receives the
  next token number for "today," with no time-slot concept and no `slotId` field in the real
  `POST /api/patient/book-appointment` request body. This deviation is honestly self-documented in
  the component's own header comment (flagged as a "SPEC DEVIATION" per
  `13-AI-Development-Rules.md`) and in `docs/15-Known-Gaps.md` Section 2.2 — it is not a hidden
  defect, but it does mean **this screen cannot be pointed at the real booking endpoint without a
  rewrite**, not just a data-wiring pass.
- The submit button reads "Book Appointment" instead of S08's canonical "Confirm Booking" string.
- A dev-only "Error Simulator" UI (a `Switch` + row of error-code buttons) is rendered directly in
  the screen — clearly commented as temporary, but currently has no build-time guard (e.g. `__DEV__`
  check) preventing it from shipping to a production bundle.

**Suggestions**

- Before this screen can be considered S08-compliant, it needs a fundamental rebuild around the
  token/queue model — not a copy-edit. Reusing `BookingWidget`'s well-built error-state handling
  (13-code mapping, bilingual copy, accessibility) is worth preserving; the slot-selection UI
  around it is not.
- Rename the submit button label to "Confirm Booking" (or pass it explicitly via
  `bookButtonLabel`) to match the canonical string.
- Wrap the Error Simulator block in a `__DEV__` conditional (or an equivalent build flag) so it
  cannot ship in a production build by accident.

## Consolidated Findings by Category

### Missing Items (highest-impact, app-wide)

1. 13 of 16 screens (81%) do not exist at all.
2. No bottom tab navigator — the entire documented navigation shell is absent.
3. No loading skeletons, empty states, or offline handling anywhere in the app.
4. 5 of 8 documented atoms, 6 of 9 molecules, 7 of 8 organisms, all layout/animation/utility
   components are unbuilt.
5. `patientsServed` is modeled but never rendered.
6. No `ErrorBoundary` — an uncaught error anywhere currently has no fallback UI.

### Incorrect Items (highest-impact)

1. **Booking screen's time-slot model contradicts the documented queue-token booking system** —
   the single most significant compliance issue in the app; self-flagged in code but still
   unresolved.
2. `DoctorCard` radius (20px) doesn't match the documented 16px, or any radius token.
3. Two Lucide icons (`Circle`, `Users`) used without being added to the Icon Library table.
4. `WaitlistForm` placed on the Home screen, a screen the documented spec never assigns it to.
5. Button/CTA label mismatches: "Book Appointment" used where "Confirm Booking" (S08) or
   "Book Appointment Now" (S07) is the canonical string.
6. Ghost button variant doesn't get its spec-mandated distinct 44px height.
7. Missing `Badge` `notification` variant.

### Suggestions (prioritized)

1. Decide the booking model question (time-slot vs. queue-token) before building any more of the
   booking flow — this is the highest-leverage fix available, since every other Booking-screen gap
   is downstream of it.
2. Build the bottom tab navigator and route groups before adding more stack screens.
3. Fix the `DoctorCard` radius value and add the two missing icons to the Icon Library table (or
   swap them for already-documented icons if a design reason exists not to expand the table).
4. Build a minimal `ErrorBoundary` and wrap the root layout with it — this is a small amount of
   code with an outsized safety benefit.
5. Treat Checkbox, Toggle, EmptyState, and LoadingSkeleton as the next atoms/molecules to build —
   they're referenced by nearly every unbuilt screen's spec and would unblock several screens at
   once.
