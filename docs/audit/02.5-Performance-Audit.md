# JivniCare Mobile — Engineering Audit, Phase 2.5: Mobile Performance Audit

**Audit type:** Read-only static analysis of rendering, memory, startup, and bundle-size
characteristics.
**Scope:** `audit-prep` branch, commit `df4085f11f8957d8cab72b98282360d0a3858b36`
**No code was modified as part of this audit.** No profiler (Flipper/React DevTools Profiler,
Xcode Instruments, Android Studio Profiler) was run — the codebase has no running build
environment configured for this audit pass, so all findings are static-analysis-derived. Where a
finding would benefit from a runtime profile, this is noted explicitly rather than presented as a
measured result.

---

## Executive Summary

At its current size (3 screens, 10 components, one hardcoded mock doctor, zero network calls),
this app has **no measurable performance problem today** — there simply isn't enough on screen or
enough data flowing through it to stress anything. The value of this audit is therefore almost
entirely **forward-looking**: identifying which currently-invisible patterns will become real
performance risks the moment real data (doctor lists, images, live queue polling) is introduced,
so they can be designed around now rather than discovered under load later. `03-Product-Vision.md`
and `04-PRD.md`'s stated V1 targets (10 doctors, 500 patients in 3 months) mean even the "at risk"
items here are unlikely to cause a visible problem for months — but several of them (image
caching, list virtualization) are far cheaper to build correctly from the start than to retrofit
after a Search Results screen already exists.

## Findings by Category

### Rendering

- **Current state:** 3 screens, each rendering a small, fixed component tree (one `DoctorCard`,
  one form, one widget). No screen currently renders more than ~15-20 native view nodes at once.
- **Risk:** 🟢 Low today. No rendering-cost problem is possible at this scale.
- **Forward risk:** 🟡 Medium — `DoctorCard` is a moderately deep tree (Avatar + 2 Badges +
  QueueStatusBadge + 2 icon rows + a nested Button, each with their own `Pressable`/`View`
  wrappers). Rendered once, this is free. Rendered 15-20× in a Search Results list (S06, not yet
  built) without virtualization, this becomes the classic "long unvirtualized list of moderately
  complex cards" problem — not a today-problem, but worth designing S06 around from the start
  rather than discovering it after the fact.

### Re-rendering

- **Current state:** No `React.memo`, no `useMemo`, no `useCallback` used anywhere for
  render-cost optimization (the few `useCallback` calls in `OTPInput.tsx` exist for stable
  prop/ref identity into child callbacks, not render-cost memoization).
- **Risk:** 🟢 Low today — component trees are shallow and state changes are localized (e.g.
  `BookingWidget`'s internal `selectedSlotId` state doesn't cause `DoctorCard` or the parent
  screen to re-render, since they're siblings, not parent/child).
- **Forward risk:** 🟡 Medium — once `DoctorCard` is rendered in a list (S05's "Doctors near you"
  or S06's results), an unmemoized `DoctorCard` will re-render every time its parent list
  re-renders (e.g. on every pull-to-refresh tick, every filter change) even for cards whose data
  hasn't changed, unless wrapped in `React.memo`. Not a defect in the current single-instance
  usage, but a pattern to establish before the list screens are built, not after.

### FlatList

- **Current state:** **No `FlatList`/`SectionList`/`@shopify/flash-list` usage exists anywhere in
  the codebase** — unsurprising, since no screen currently renders more than one card (`app/index.tsx`
  maps zero arrays; there is exactly one `<DoctorCard>` JSX element in the entire app). Lists are
  rendered, where they exist at all (e.g. the dev-only error-code button row in the Booking
  screen), via a plain `.map()` inside a `View`.
- **Risk:** 🟢 Low today (nothing to virtualize).
- **Forward risk:** 🔴 **High** — `docs/07-Mobile-UX-Spec.md` documents 4 list-heavy screens (S06
  Search Results — infinite scroll, 15/page; S10 My Bookings — full unbounded history per
  `docs/11-API-Contract.md`'s confirmed no-pagination behavior; S15 Notification Inbox). None of
  these exist yet, so there is no _current_ virtualization defect — but `docs/12-Backend-Spec.md`
  confirms `GET /api/patient/my-bookings` returns a patient's **entire** booking history in one
  unbounded response with no server-side pagination. A long-time active user's My Bookings screen,
  if built with a plain `.map()` the way the current codebase's only precedent (the error-code
  button row) does it, would render every booking as a native view simultaneously — this is the
  single highest-leverage performance decision available to make correctly before any list screen
  is built, precisely because the backend has already foreclosed pagination as a mitigation.

### Memoization

- Already covered above (Re-rendering). No memoization exists; none is needed yet; needed soon.

### Images

- **Current state:** `Avatar.tsx` uses React Native's base `<Image source={{uri: source}} />` —
  no `expo-image`, no disk/memory caching layer, no placeholder/blurhash, no explicit
  `cache-control` handling. `doctor.profilePhoto` is `undefined` in the only mock data record that
  exists today, so this code path has never actually been exercised with a real remote image.
- **Risk:** 🟢 Low today (no real images are ever loaded).
- **Forward risk:** 🟠 **High** — the moment `profilePhoto` URLs are wired to real doctor photos
  (S05/S06/S07 all render an `Avatar` per doctor), base `Image` will re-fetch from network on every
  mount with no memory/disk cache reuse across screens, and no lazy/progressive loading — directly
  contradicting `docs/04-PRD.md` F05's own acceptance criterion ("images lazy-loaded"). This is
  the most concrete, already-documented-elsewhere performance requirement the codebase does not
  yet meet, because the underlying feature (real photos) doesn't exist yet either.
- Clinic photo sliders (S07, up to 3 images, swipeable) are entirely unbuilt — no additional image
  work exists to evaluate there yet.

### Fonts

- **Current state:** No custom font loading anywhere (no `expo-font` dependency, no `useFonts`
  call, no font asset files in `assets/`). The app relies entirely on system fonts (San
  Francisco/Roboto), matching `docs/08-Design-System.md`'s Typography section's stated primary
  choice exactly.
- **Risk:** 🟢 None. This is a deliberate, spec-compliant decision, not a gap — system fonts avoid
  the startup-time font-loading flash/delay that a custom-font app would need to handle (e.g. via
  `SplashScreen.preventAutoHideAsync()` gating). No action needed unless a future design decision
  introduces a custom font (the doc's own "Fallback: Inter (if custom font loaded)" language
  implies this was considered and deliberately not exercised for V1).

### Bundle Size

- **Current state:** `package.json` lists a lean 14 runtime dependencies — no heavyweight
  incidental libraries (no lodash, no moment.js, no large UI kit). `lucide-react-native` is
  tree-shakeable (confirmed by its own description in `docs/08-Design-System.md`), and only ~15
  distinct icon names are imported across the whole codebase, so its actual bundle contribution is
  small regardless of the package's total icon count.
- **Risk:** 🟢 Low today.
- **Forward risk:** 🟡 Medium — `react-native-reanimated` (^4.5.2) and `react-native-worklets`
  (^0.11.1) are both installed but **entirely unused** (confirmed: `OTPInput.tsx`'s shake animation
  uses React Native's core `Animated` API, not Reanimated). These are not small dependencies —
  Reanimated in particular adds a native module + a Babel plugin step to every build. Carrying them
  unused doesn't bloat the _JS_ bundle much (tree-shaking should drop unused exports), but it does
  add native binary size and build-time cost for zero current benefit. Either start using them
  (there are several documented `FadeIn`/`SlideUp`/`Pulse` animation components that would be a
  natural fit) or defer adding them until they're needed.
- No bundle analyzer (`expo-atlas`, `source-map-explorer`, or similar) has been run — this audit
  cannot report actual byte-size numbers, only dependency-count-level risk.

### Navigation

- **Current state:** A single flat Expo Router `<Stack>` with 3 screens registered. No route
  groups, no lazy-loaded route segments beyond Expo Router's own default per-route code-splitting
  behavior (each `app/*.tsx` file is already its own bundle chunk by the framework's file-based
  routing convention — this is "free" lazy loading the app already benefits from without any
  explicit work).
- **Risk:** 🟢 Low today — 3 screens is too small a stack for navigation performance to matter.
- **Forward risk:** 🟢 Low — Expo Router's file-based routing means growing to 16 screens doesn't,
  by itself, introduce a navigation performance problem; the earlier-flagged _architectural_ gap
  (no route groups, no auth-gating layout) is a maintainability/security concern
  (`02.1-Architecture-Audit.md` Section 3), not a performance one.

### Animation

- **Current state:** Exactly one animation exists in the entire app: `OTPInput.tsx`'s shake, built
  with React Native's core `Animated` API (not Reanimated), and correctly gated behind
  `AccessibilityInfo.isReduceMotionEnabled()`.
- **Risk:** 🟢 None — a single, simple, correctly-implemented animation has no measurable
  performance cost.
- **Forward risk:** 🟢 Low, contingent on tooling choice — `docs/09-Component-Library.md` Section 5
  documents 4 animation wrapper components (`FadeIn`, `SlideUp`, `Shake`, `Pulse`) intended to run
  on the native thread. If these are eventually built using core `Animated` (JS-thread-driven)
  instead of the already-installed Reanimated (native-thread-driven), animations on low-end Android
  devices (`docs/06-User-Personas.md`'s target hardware, ₹8-15K phones) could visibly stutter under
  JS-thread contention — a real risk given the target user base, but entirely avoidable by using
  the dependency that's already installed for exactly this purpose.

### Memory

- **Current state:** No large in-memory data structures exist (one mock doctor object, a handful
  of small form-state strings). No obvious retained-reference leaks found in a manual read of every
  component's cleanup behavior, with two specific exceptions (see below).
- **Risk:** 🟢 Low today.
- **Findings carried over from the Code Quality Audit (`02.4`), restated here in performance
  terms:**
  - `app/index.tsx`'s `handleWaitlistSubmit` and `app/booking/[id].tsx`'s `handleBookingSubmit`
    both fire uncancelled `setTimeout`-based state updates with no unmount guard. At today's scale
    this leaks nothing meaningful (a already-GC'd closure firing a few state setters against an
    unmounted Fiber is a warning, not a real memory retention problem in React 19) — but it is the
    exact pattern that, multiplied across dozens of real network calls once F08/F12 are wired to
    real endpoints, becomes a genuine source of subtle bugs and wasted work under real network
    latency (a user backing out of a slow booking request shouldn't leave a dangling timer/promise
    behind).
- No `WeakMap`/cache/subscription-registry code exists yet that could leak (no FCM listeners, no
  polling intervals, no WebSocket/SSE clients) — this section is almost entirely forward-looking
  because almost none of the memory-relevant _feature_ code (Section 9/F09 Token Tracking's 30s
  polling, F16's FCM listeners) has been built yet.

### Startup

- **Current state:** No `expo-splash-screen` custom configuration found (no `SplashScreen.preventAutoHideAsync()`
  call, no explicit splash-hide-on-ready gating) beyond Expo's own default managed splash. No
  heavy synchronous work happens at module-load time in any screen (all mock data is a small
  static object, not a large JSON parse).
- **Risk:** 🟢 Low today — a 3-screen, zero-network-call app has close to the fastest possible
  cold-start profile achievable in Expo/React Native; there is nothing to measure a regression
  against yet.
- **Forward risk:** 🟡 Medium — `docs/04-PRD.md` Section 7's Success Criteria explicitly targets
  "Cold start <3s." No startup-time instrumentation (e.g. a simple `Date.now()` delta logged once
  at first-interactive, or a Firebase Performance Monitoring trace) exists to verify this target
  once real initialization work (auth token restore from Keychain/Keystore, language-preference
  read, initial doctor-search prefetch) is added to the startup path. Recommend adding a startup
  timer now, even as a no-op measurement, so there's a baseline before real work is added to
  compare against.

### Lazy Loading

- **Current state:** No explicit `React.lazy`/dynamic `import()` usage anywhere — not needed yet,
  since Expo Router's file-based routing already lazy-loads each screen's module by convention,
  and there are only 3 screens.
- **Risk:** 🟢 Low today.
- **Forward risk:** 🟢 Low — this will remain a non-issue as long as new screens continue to be
  added as new `app/*.tsx` files (the framework default), rather than all being imported eagerly
  into one mega-component. No action needed now; worth re-checking once S06's `FilterPanel`
  bottom sheet and similarly heavy, conditionally-rendered UI exist, to confirm they aren't
  imported/constructed until actually opened.

### Future Backend Performance

This section evaluates what the _mobile client_ will need to do once real backend calls replace
today's mocks — not the backend's own performance (out of scope for a mobile-code audit), but the
client-side patterns that determine whether backend performance problems become user-visible.

| Risk                                                                                                                                                                                  | Severity                                                  | Why                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `GET /api/patient/my-bookings` returns a full, unbounded booking history with no pagination (confirmed in `docs/11-API-Contract.md`/`docs/12-Backend-Spec.md`)                        | 🔴 High                                                   | The mobile client will need to either virtualize the resulting list (see FlatList section) or risk both a slow JSON parse and a slow render for any patient with a long history — there is no server-side mitigation available, so the client bears 100% of the responsibility here                                                                                                                                                                                                                                                                  |
| `GET/POST /api/v1/patient/search` caps at 100 raw DB matches before scoring                                                                                                           | 🟡 Medium                                                 | Not a mobile-code risk directly, but the client should not assume "no results shown beyond position 100" means "no more doctors exist" — worth designing the empty/end-of-list UI copy around this known ceiling now rather than after a confused support ticket                                                                                                                                                                                                                                                                                     |
| `POST /api/patient/book-appointment` and 6 other patient routes are cookie-only (Bearer auth not yet wired, per `11-API-Contract.md`'s Blocker 1 / `14-Feature-Dependencies.md` P0-1) | 🟡 Medium (performance-adjacent, not itself a perf issue) | Once P0-1 lands, the mobile client's retry/backoff behavior on 401s (e.g. from an expired 30-minute access token needing a refresh-token round-trip) needs to be designed to avoid a double-network-round-trip on every single request post-expiry — worth designing the auth-refresh interceptor with this in mind rather than bolting it on reactively                                                                                                                                                                                             |
| F09 Token Tracking's real-time requirement (SSE today, FCM-mandated for mobile)                                                                                                       | 🟡 Medium                                                 | `docs/04-PRD.md` F06/F09 already correctly identify that SSE is unsuitable for a backgrounded mobile app; if a _polling_ fallback is built instead of FCM (explicitly warned against in `14-Feature-Dependencies.md`'s "not safe to parallelize" notes), an aggressive fixed-interval poll (e.g. every 1-2s instead of the documented 30s) could become a real battery/network-cost problem on the target low-end Android hardware — no code exists yet to evaluate, flagged purely as a forward risk given the explicit warning already in the docs |

## Recommendations (priority order)

1. **Before building S06 (Search Results) or S10 (My Bookings):** commit to `FlatList` (or
   `@shopify/flash-list` for the heavier `DoctorCard` rows) as the standard list primitive, and
   wrap `DoctorCard` in `React.memo` at the same time — these two changes together address the
   single largest forward-looking risk in this report and are cheapest to do together, from the
   start, rather than sequentially after a performance complaint.
2. **Before wiring real `profilePhoto` URLs:** replace `Avatar.tsx`'s base `Image` with `expo-image`
   (a near-drop-in replacement) to get memory/disk caching and placeholder support for free —
   directly satisfies `docs/04-PRD.md` F05's already-documented "images lazy-loaded" requirement.
3. **Decide the Reanimated question now, not later:** either build the documented `FadeIn`/
   `SlideUp`/`Pulse` components against the already-installed `react-native-reanimated` (native
   thread, appropriate for the target low-end Android hardware), or remove the two unused
   dependencies until they're actually needed — don't let them sit installed-but-idle indefinitely.
4. Fix the two unmount-safety gaps flagged in the Code Quality Audit (`02.4`) — small effort, and
   directly relevant here as the template for how future real network calls should be cancelled.
5. Add a simple startup-time measurement now (even a `console`-free, Firebase-Performance-style
   trace) so there's a documented baseline to compare against once real initialization work lands.
6. When F09's real-time layer is built, follow `14-Feature-Dependencies.md`'s own explicit
   guidance and build it against FCM from Phase 3, not a polling placeholder — the docs already
   correctly flag this; this audit's job is only to reinforce that the performance risk of not
   doing so (battery/network cost on low-end hardware) is real, not just an architectural nicety.
