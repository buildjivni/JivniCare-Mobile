# JivniCare Mobile — Engineering Audit, Phase 2.4: Production-Grade Code Quality Audit

**Audit type:** Read-only static analysis of every file in `app/` and `src/`.
**Scope:** `audit-prep` branch, commit `df4085f11f8957d8cab72b98282360d0a3858b36`
**No code was modified as part of this audit.** No lint/test tooling exists in this repository
(confirmed — see Section "Tooling Gaps" below), so this audit was performed entirely by direct
line-by-line reading of all 14 source files, cross-referenced against `tsconfig.json`'s settings.

---

## Executive Summary

The ~1,600 lines of application code in this repository are, line-for-line, unusually disciplined
for a pre-lint, pre-test codebase: strict TypeScript throughout, zero `any`, zero `console.*`
calls, no dead files, and a consistent naming convention. The quality problems found are almost
all **absences** (no lint config, no tests, no error boundary) or **small, contained issues**
(a handful of duplicated helper functions, two unmount-safety gaps, one comment that documents
behavior the code doesn't actually implement) rather than systemic code smells. There is no
evidence of circular dependencies, memory leaks beyond two minor unmount-safety gaps, or
uncontrolled logging.

## Findings by Category

### 1. TypeScript

**Grade: Excellent.**

- `tsconfig.json` has `"strict": true`. Confirmed zero compile errors project-wide.
- Zero occurrences of `any` or `as any` anywhere in `src/` (verified via full-text search).
- Every component prop, every callback, every exported helper has an explicit type or a correctly
  inferred one — no implicit `any` parameters found.
- Discriminated unions used correctly (`BookingErrorCode`, `OtpVerifyErrorCode`,
  `QueueStatusBadgeStatus` are all closed string-literal unions, not loose `string`).

No issues found in this category.

### 2. Naming

**Grade: Good, one internal inconsistency.**

- Component names, prop names, and helper function names are consistently `PascalCase`/`camelCase`
  and descriptive (`initialsFromName`, `formatFee`, `mapVerifyOtpApiError`, `isPhoneValid`).
- 🟡 **Medium** — Two unrelated concepts share the identifier `BookingWidget`: the built molecule
  (`src/components/molecules/BookingWidget.tsx`, slot-based) and the documented organism
  (`docs/09-Component-Library.md` Section 3.4, token-based). This is a naming collision between
  code and docs, already self-flagged in `docs/15-Known-Gaps.md` Section 2.2, but it remains
  unresolved and is a real risk for a future developer who searches for "BookingWidget" and finds
  the wrong contract.

### 3. Folder Organization

**Grade: Good for what exists, incomplete for what's needed.**

- Atomic Design folder split (`atoms/molecules/organisms`) is real and consistently followed — no
  layering violations found (see `02.1-Architecture-Audit.md` Section 7 for the full layering
  verification).
- 🟡 **Medium** — No `hooks/`, `services/`, `utils/`, `constants/`, or `store/` folders exist,
  meaning there is currently no obvious "where does this go" answer for non-component code. (Full
  detail already covered in `02.1-Architecture-Audit.md` Sections 1, 8, 10, 11 — not re-litigated
  here in depth to avoid duplicating that report; referenced because it is also, correctly, a code
  _organization_ quality issue.)

### 6. Component Duplication

- 🟠 **High** — `BookingWidget` exists as two conceptually different components under the same
  name (see Naming section above). This is component duplication in the sense that matters most:
  a future integrator has a 50/50 chance of finding the wrong one first.
- No other component duplication found — the atoms/molecules/organisms inventory has no two
  components solving the same problem twice.

### 7. Dead Code

- No dead files, no unreachable branches, no commented-out code blocks found in any of the 14
  source files. This is a clean codebase in the "no zombie code" sense — everything present is
  reachable and (mostly) used.
- The one exception is the `patientsServed` field noted under Unused Code above — "dead" in the
  sense of being written but never read, though it's a data field, not a code path.

### 8. Circular Dependency

- **None found.** Verified by reading every `import` statement in all 14 files: atoms import only
  React Native/Lucide/shared types; molecules import only atoms (+ one sibling molecule,
  `DoctorCard` → `QueueStatusBadge`, an acceptable molecule-composes-molecule pattern); the one
  organism imports only atoms; screens (`app/`) import only from `src/`, never the reverse. No
  import graph cycle is possible with this structure.

### 9. Memory Leaks

| Finding                                                                                                                                                                                                                                                                                                                                | Severity      | Location                     |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- | ---------------------------- |
| `handleWaitlistSubmit`'s `setTimeout(() => { setLoading(false); setSuccess(true); setPosition(4); }, 800)` has no cleanup — if the Home screen unmounts (user navigates away) before 800ms elapses, this fires 3 state setters on an unmounted component (React will log a warning; in React 19 this is non-fatal but still incorrect) | 🟡 **Medium** | `app/index.tsx:31-35`        |
| `handleBookingSubmit`'s `await new Promise((resolve) => setTimeout(resolve, 600))` has the same unmount-safety gap — a fast back-navigation during the simulated network delay will attempt to call `setConfirmedSlotLabel` after unmount                                                                                              | 🟡 **Medium** | `app/booking/[id].tsx:44-45` |
| All `useEffect` hooks in the codebase (`OTPInput.tsx` ×3, `OTPInputBox.tsx` ×2, `WaitlistForm.tsx` ×1) were individually checked: **none of them create timers, subscriptions, or listeners that would need a cleanup function** — they only synchronize local state from props. No leak risk here.                                    | ✅ N/A        | Verified clean               |

Both findings above are low-_impact_ today (mock data, short delays, small blast radius) but are
exactly the pattern that becomes a real bug once real network latency (multi-second, on 3G) is
introduced — worth fixing now while the fix is trivial (an `isMounted`/`AbortController` guard or
a cleanup-returning effect) rather than after real API calls make it likelier to actually fire.

### 10. Async Handling

- `handleBookingSubmit` is correctly `async`/`await`, and `BookingWidget` correctly awaits it and
  catches/displays the thrown `BookingSubmitError` — this is good, idiomatic error propagation.
- 🟡 **Medium** — Neither of the two `setTimeout`-based mocks (Home, Booking) uses
  `AbortController` or an equivalent cancellation mechanism — tied to the Memory Leaks finding
  above; the same fix addresses both.
- No unhandled promise rejections found — every `.then`/`await` site has a corresponding
  `.catch`/`try-catch` (`BookingWidget`'s submit handler wraps its await in try/catch and routes
  the error through `getBookingErrorMessage`).

### 11. Error Handling

**Grade: Strong where it exists, absent at the app-shell level.**

- `BookingWidget`'s error handling is a genuine strength: the full 13-named-code B6 taxonomy is
  modeled, mapped to bilingual-source copy, and displayed with correct `accessibilityLiveRegion="assertive"`
  semantics.
- 🔴 **High** — **No `ErrorBoundary` exists anywhere in the app**, and `app/_layout.tsx` does not
  wrap its `<Stack>` in one. A single uncaught render-time error in any component (a `null` access,
  a bad prop) will currently crash the entire app to Expo's default red/white error screen with
  no recovery path, contradicting `09-Component-Library.md` Section 6.1's documented
  `ErrorBoundary` contract (`"Something went wrong" + [Retry]`) — this component simply doesn't
  exist yet.
- 🟡 **Medium** — No network-level error handling exists yet (no fetch/axios client to wrap), but
  this is a consequence of no networking code existing yet at all, not a defect in existing code.

### 12. Logging

- **Zero `console.log`/`console.warn`/`console.error` calls found anywhere in `src/` or `app/`**
  (verified via full-text search). This is unusually clean — no debug-logging cruft left behind.
- 🟡 **Medium** (forward-looking, not a current defect) — there is also no structured logging
  strategy (e.g. no Sentry/Bugsnag/analytics event wrapper) established yet for when real errors
  need to be captured in production. Worth deciding before the first real API integration lands,
  since retrofitting logging around already-written `try/catch` blocks is easy, but only if
  someone remembers to do it.

### 13. Constants

- 🟡 **Medium** — Several meaningful values are module-local constants rather than shared,
  discoverable ones: `MOCK_SUBMIT_DELAY_MS` (booking screen), `PHONE_DIGIT_COUNT`/`MIN_NAME_LENGTH`
  (WaitlistForm), `DEFAULT_LENGTH` (OTPInput). None of these are wrong, but none are in a shared
  `src/constants/` location either, consistent with `02.1-Architecture-Audit.md`'s Section 11
  finding.
- ✅ `BOOKING_ERROR_CODES` is a well-formed, correctly exported constant array explicitly built
  "for exhaustiveness checks" — a good example of doing this right when it matters.

### 16. Code Smell

- 🟡 **Medium** — **Comment/implementation drift**: `Button.tsx`'s own comment (line 22) states
  "44px for ghost," but `HEIGHT_CLASS` has no `ghost`-specific override — the `ghost` variant's
  height is driven entirely by the generic `size` prop like every other variant, meaning the
  documented-in-comment 44px behavior is never actually enforced. A comment describing behavior
  the code doesn't implement is a more dangerous smell than no comment at all, since it actively
  misleads the next reader.
- 🟢 **Low** — The dev-only "Error Simulator" UI in `app/booking/[id].tsx` (a `Switch` + 13
  error-code buttons) is not gated behind `__DEV__` or any build-time flag, meaning it will render
  in a production build unless removed manually before shipping — low severity today (there is no
  release build being cut yet) but worth a `__DEV__` guard now while it's a one-line fix, rather
  than relying on someone remembering to delete the block later.
- 🟢 **Low** — `formatErrorCodeLabel` in `app/booking/[id].tsx` re-implements a generic
  "SNAKE_CASE → Title Case" string transform inline in a screen file rather than as a shared
  utility — low impact today (one call site) but the same "utility function trapped in a screen
  file" pattern flagged elsewhere in this audit set.

### 17. Maintainability

**Grade: Good, with the caveats above.**

- Every deviation-from-spec found across this entire audit set was, on inspection, _already_
  self-documented in a code comment and/or `docs/15-Known-Gaps.md` — this is a genuinely strong,
  unusual-for-this-project-stage maintainability trait: a future maintainer is very unlikely to be
  surprised by something the current code already knows about and has written down.
- The main maintainability risk is not the code that exists, but the tooling gap: without ESLint,
  a test suite, or CI, every finding in this report will need to be caught by a human reading the
  diff on every future change, indefinitely, rather than by automation.

**Scoring method:** Technical Debt Score weights breadth-and-depth of unresolved issues (higher =
worse); Overall Code Quality Score weights current code health independent of unbuilt features
(higher = better) — these are intentionally different axes: a codebase can have low debt (little
accumulated cruft) while still being incomplete (most features unbuilt), which is exactly this
codebase's situation. Feature completeness is scored separately in
`02.1-Architecture-Audit.md` and `02.3-Documentation-Traceability-Matrix.md` — this report scores
_the code that exists_, not how much of the product exists yet.

## Consolidated Recommendations (priority order)

1. Add ESLint (with the React/React Native + `@typescript-eslint` presets) and a minimal Jest +
   `@testing-library/react-native` setup before writing significantly more code — every finding in
   this report except the bilingual-strings gap would be easier to prevent from recurring with
   basic lint rules alone (`no-console`, `no-duplicate-imports`, a custom rule or code-review
   checklist item for raw hex colors outside a theme file).
2. Extract `src/theme/colors.ts` (named JS constants for the hex values already duplicated across
   files) and `src/utils/phone.ts` (single shared phone validation/masking implementation) — both
   are small, mechanical, and immediately remove 2 of the 3 largest "Duplicate Code" findings.
3. Add a minimal `ErrorBoundary` component and wrap the root `_layout.tsx` `<Stack>` in it.
4. Fix `DoctorCard`'s radius to 16px (or update the doc if 20px is actually the intended final
   value — either way, the current silent mismatch should be resolved one way, explicitly).
5. Add unmount-safety guards (or replace the two `setTimeout` mocks with a small shared
   `mockDelay(ms, { signal })` helper using `AbortController`) to the Home and Booking screens.
6. Render `doctor.patientsServed` somewhere, or remove the field until a screen actually needs it
   — don't leave a modeled-but-unused field as the status quo.
7. Resolve the `BookingWidget` naming collision with the documented organism — rename one of them.
