# JivniCare Mobile — Engineering Audit, Phase 2.1: Architecture Audit

**Audit type:** Read-only architecture, implementation, and documentation review
**Scope:** `audit-prep` branch, commit `df4085f11f8957d8cab72b98282360d0a3858b36`
**Method:** Full read of every file in `app/`, `src/`, root config files, and `docs/`
**No code was modified as part of this audit.**

---

## Executive Summary

The repository is a **very early-stage MVP scaffold**, not a fully-layered application. It
contains 3 routed screens, a genuinely well-built atomic-design component library (atoms →
molecules → organisms), and exceptional supporting product/design documentation (15 locked
docs). However, almost every "backend-facing" or cross-cutting architectural layer a mobile app
of this ambition will eventually need — state management, API/service layer, hooks, utilities,
constants, theming provider, navigation guards — **does not exist yet**, or exists only as an
unused dependency (`zustand` is installed but has zero usages anywhere in `src/` or `app/`).

This is not a criticism of intent — the documentation set makes clear this is a deliberate
"frontend-first, pre-backend-integration" build, and the code that *does* exist is disciplined
(strict TypeScript, consistent accessibility props, deviations from spec are flagged in comments
rather than silently guessed at). The risk is that the *absence* of these layers is currently
invisible to anyone who hasn't read every file — this report makes that explicit.

## 1. Project Structure

### Current State
```
app/                          # Expo Router screens (4 files)
  _layout.tsx                 # Root Stack, no route groups
  index.tsx                   # Home
  doctor/[id].tsx              # Doctor profile
  booking/[id].tsx              # Booking
src/
  components/
    atoms/        (5 components: Avatar, Badge, Button, Input, OTPInput)
    molecules/    (4 components: BookingWidget, DoctorCard, OTPInputBox, QueueStatusBadge)
    organisms/    (1 component: WaitlistForm)
  data/            (1 file: mockDoctor.ts)
  types/           (1 file: accessibility.ts)
assets/brand/       # SVG/PNG brand assets only — no app-specific UI imagery
docs/               # 15 locked product/design/API documents
```
14 source files total (4 screens + 10 components + 1 mock data + 1 types file).

### Strengths
- Clean top-level split between routing (`app/`) and implementation (`src/`) — the Expo Router
  convention is followed correctly (screens are thin, logic/markup delegated to `src/components`).
- Atomic-design folder split (`atoms/molecules/organisms`) is real, not just aspirational — every
  molecule composes only atoms/lower molecules, every organism composes molecules/atoms. No
  layering violations found.
- Each component folder has an `index.ts` barrel export — consistent import ergonomics
  (`@/components/molecules` rather than deep relative paths).

### Weaknesses
- No `templates/` layer despite Atomic Design being the stated pattern (see Section 7).
- No `hooks/`, `services/`, `utils/`, `constants/`, `lib/`, `store/`, or `api/` folders exist
  **at all** — confirmed via exhaustive glob search. Every cross-cutting concern currently lives
  inline inside screen files (e.g. `BookingScreen`'s error-simulator state, `formatFee`/
  `initialsFromName` helpers duplicated inline in `DoctorCard.tsx` rather than in a shared
  `utils/`).
- `src/data/mockDoctor.ts` is a single hardcoded doctor object — there's no mock-data folder
  structure (e.g. per-entity mock files, a mock API layer) that would scale past one demo record.

### Risks
- **Medium–High**: As more screens are added, without a `utils/`/`hooks`/`constants` convention
  established *now*, formatting helpers (`formatFee`, phone masking, date formatting) will keep
  getting re-implemented per-component (already happening: `formatFee` in `DoctorCard.tsx`,
  `maskPhone` in `OTPInputBox.tsx`, `digitsOnly`/`isPhoneValid` in `WaitlistForm.tsx` — three
  separate phone-related helpers already, none shared).

### Recommendations
- Introduce `src/utils/` (formatting, validation) and `src/constants/` (route names, config
  values) folders before the next feature pass, and migrate the three duplicated phone/format
  helpers into them as the first inhabitants.
- Decide now whether Atomic Design's `templates` layer will be used, and if so scaffold an empty
  `src/components/templates/` folder with a README stub so the convention is visible before it's
  needed under time pressure.

## 2. Clean Architecture

### Current State
There is effectively **one layer**: Presentation. Screens (`app/`) call components
(`src/components/`) which render UI and hold local `useState`. There is no Business Layer, no
Data Layer, no Service Layer, and no Repository Pattern anywhere in the codebase.

| Clean Architecture Layer | Found? | Evidence |
|---|---|---|
| Presentation | ✅ Yes | `app/*.tsx`, `src/components/**` |
| Business/Domain Layer | ❌ No | No domain logic module; validation lives inline in components (e.g. `isPhoneValid` inside `WaitlistForm.tsx`) |
| Data Layer | ⚠️ Placeholder only | `src/data/mockDoctor.ts` — a static constant, not a data-access abstraction |
| Service Layer | ❌ No | No network client, no `fetch`/`axios` usage anywhere (confirmed via search — zero `process.env`/`EXPO_PUBLIC_` references, zero network calls) |
| Repository Pattern | ❌ No | Nothing to abstract yet — screens import `MOCK_DOCTOR` directly |
| Dependency Direction | ✅ Correct so far | Components never import from `app/`; `app/` imports from `src/` — the one dependency direction that exists is correct |

### Strengths
- The one dependency rule that *is* enforced (screens depend on components, never the reverse) is
  followed with no violations found in any of the 4 screens or 10 components.
- Business-rule *documentation* (13 named booking error codes, OTP failure codes, waitlist
  validation rules) is unusually rigorous for this project stage — see `BookingWidget.tsx`'s
  `BOOKING_ERROR_CODES` and `OTPInputBox.tsx`'s `mapVerifyOtpApiError`. The domain logic exists as
  *typed constants and pure mapping functions*, which is a very migratable starting point for a
  future domain layer — it just isn't organized into one yet.

### Weaknesses
- Validation logic (`isNameValid`, `isPhoneValid` in `WaitlistForm.tsx`) is private to one
  component file and not reusable by any future screen needing the same phone/name validation
  (e.g. a future Login screen).
- `mapVerifyOtpApiError` (in `OTPInputBox.tsx`) and `getBookingErrorMessage` (in
  `BookingWidget.tsx`) are genuinely reusable pure functions already — but they live inside UI
  component files instead of a `src/services/` or `src/domain/` module, so they're only
  discoverable by reading component internals.
- No repository/data-access abstraction means every screen currently imports `MOCK_DOCTOR`
  directly by name — swapping to a real API later requires touching every screen file rather than
  one data-access module.

### Risks
- **High**: Without a Service/Repository layer established before real backend integration
  begins, the natural path of least resistance is to `fetch()` directly inside screen components
  — which is exactly the anti-pattern Clean Architecture exists to prevent, and which the
  project's own `docs/15-Known-Gaps.md` implies is still an open decision ("whichever screen
  wires the real endpoint" language appears repeatedly in code comments across `WaitlistForm.tsx`,
  `BookingWidget.tsx`, `OTPInputBox.tsx`).

### Recommendations
- Before any real endpoint is wired (blocked anyway on backend gaps per `docs/15-Known-Gaps.md`
  Section 2), create `src/services/api/` with a single HTTP client wrapper and one
  repository-style module per resource (`doctorRepository.ts`, `bookingRepository.ts`,
  `waitlistRepository.ts`) that screens call instead of importing mock data directly.
- Extract `isNameValid`/`isPhoneValid`/`mapVerifyOtpApiError`/`getBookingErrorMessage` into a
  `src/domain/` or `src/utils/validation.ts` module now, while there are still few call sites —
  this is a pure, low-risk move that pays off the moment a second screen needs phone validation.

## 3. Navigation

### Current State
Expo Router (file-based), single root `<Stack>` in `app/_layout.tsx`, 3 screens registered:
`index`, `doctor/[id]`, `booking/[id]`. No route groups (`(tabs)`, `(auth)`, etc.) exist.

### Strengths
- Correct use of Expo Router dynamic segments (`[id].tsx`) for both doctor profile and booking
  screens, with `useLocalSearchParams<{ id: string }>()` typed correctly in both.
- Centralized header theming in `_layout.tsx` (colors, title styles) rather than repeated
  per-screen `options` — good practice, matches `docs/08-Design-System.md`'s header spec exactly
  (colors cited inline in code comments).
- Navigation calls (`router.push`) are simple, correctly typed template-literal paths.

### Weaknesses
- **No route groups at all.** Everything is a flat, ungrouped stack. There's no `(auth)` group
  for pre-login screens, no `(tabs)` group for a future bottom-tab home experience (Home/Search/
  Bookings/Profile, which `docs/07-Mobile-UX-Spec.md`'s 16-screen inventory implies is coming).
- **No protected routes.** There is no auth-gating mechanism anywhere — no redirect-if-unauthenticated
  logic, no `Slot`/layout guard pattern. This is consistent with there being no auth state at all
  yet (see Section 4), but it means the navigation layer has no seam prepared for when login is
  added.
- Both dynamic-route screens (`doctor/[id].tsx`, `booking/[id].tsx`) read `id` but **ignore it** —
  both hardcode `MOCK_DOCTOR` regardless of the route param (explicitly commented as such in
  `doctor/[id].tsx`: "always renders MOCK_DOCTOR regardless of `id`"). This is honestly
  self-documented, but means the routing layer is currently unverified against real multi-doctor
  navigation.

### Risks
- **Medium**: `docs/07-Mobile-UX-Spec.md` describes 16 screens; only 3 exist. Retrofitting route
  groups (especially an `(auth)` gate) after more screens accumulate in a flat structure is a
  larger, riskier change than establishing the group structure now with only 3 screens to move.
- **Low** (currently): the `id`-ignored pattern is low-risk today (single mock doctor) but is a
  landmine if a second doctor is added to mock data before the real lookup is wired — a developer
  could reasonably assume the id param already works.

### Recommendations
- Introduce `(auth)` and `(app)` (or similar) route groups now, even with only one screen in each,
  so the auth-gating layout seam exists before login is built.
- Add a tracking comment or `docs/15-Known-Gaps.md` entry (pattern already used elsewhere in this
  repo) noting that `[id]` params are currently decorative, so nobody adds a second mock doctor
  assuming routing already discriminates between them.

## 4. State Management

### Current State
**No global state management is in use anywhere in the codebase**, despite `zustand@^5.0.14`
being declared in `package.json`'s dependencies. A repo-wide search for `zustand` imports or
`create(` store calls returned zero matches. No `Context` providers exist. No React Query /
TanStack Query dependency exists. All state found is local `useState` inside individual screens
and components (e.g. `BookingScreen`'s `confirmedSlotLabel`/`simulateError` state,
`WaitlistForm`'s internal `attempted` flag).

### Strengths
- Where local state is used, it's used correctly and minimally — components favor controlled
  props (`value`/`onChange`) over owning state they shouldn't (e.g. `WaitlistForm` takes `name`/
  `phone` as controlled props from the parent screen rather than owning them internally, keeping
  state ownership at the screen level where it will eventually need to sync with a store/API).
- State ownership boundaries are clear and intentional per-component: `BookingWidget` owns its own
  `selectedSlotId`/`status` (purely presentational, appropriately local), while `WaitlistForm`
  correctly pushes `name`/`phone`/`loading`/`success` up to the parent screen.

### Weaknesses
- **Zustand is dead weight right now** — an unused dependency increases install size and gives a
  false signal (via `package.json`) that state management architecture exists when it does not.
- No pattern is established for *where* global state (auth session, selected language Hindi/
  English — referenced throughout component docs-comments as "sourced from the app's language
  state" — cart/booking-in-progress state) will eventually live. `docs/09-Component-Library.md`'s
  Section 0 is referenced by multiple components as the source for a language-state concept that
  doesn't exist in code yet.
- No React Query (or equivalent) exists for server-state caching, which will matter significantly
  once real API calls replace the current mock `Promise`/`setTimeout` simulations in
  `BookingWidget.handleBookPress` and `home/index.tsx`'s `handleWaitlistSubmit`.

### Risks
- **Medium**: The repeated code-comment pattern "sourced from the app's language state, not
  decided by this component" (found in `BookingWidget.tsx`, implied elsewhere) means multiple
  components already have an *implicit contract* with a global language store that doesn't exist
  yet — when it's finally built, whoever builds it needs to satisfy several already-written
  implicit contracts simultaneously, which is a coordination risk if not tracked.
- **Low**: the unused Zustand dependency itself is not a runtime risk, just a maintenance/audit
  clarity issue.

### Recommendations
- Either scaffold a minimal `src/store/` with a first real Zustand store (even just an
  `useLanguageStore` matching the implicit contract already referenced in component comments) or
  remove the unused dependency until it's actually needed — don't leave it installed-but-silent.
- Evaluate adding `@tanstack/react-query` (or an equivalent) before real API integration begins,
  given the number of Known-Gaps items describing pending network calls.
- Add one line to `docs/15-Known-Gaps.md` explicitly tracking "global language state" as a
  cross-cutting item several components already assume exists, so it isn't rediscovered
  piecemeal per-component.

## 5. API Layer

### Current State
There is no API layer. Zero network calls exist in the codebase (confirmed by searching for
`fetch`, `axios`, `process.env`, `EXPO_PUBLIC_` — no matches anywhere). All three screens that
need data use imported mock constants or inline mock arrays:
- `app/index.tsx` / `app/doctor/[id].tsx`: import `MOCK_DOCTOR` from `src/data/mockDoctor.ts`
- `app/booking/[id].tsx`: defines `MOCK_SLOTS` inline in the screen file itself (not even in
  `src/data/`)

"Submission" flows are simulated with `setTimeout`/`Promise` delays that always succeed (Home's
waitlist) or are toggled by a manual dev-only error simulator switch (Booking screen).

### Strengths
- The mock/real seam is already anticipated at the *component* level in a genuinely good way:
  `WaitlistForm`, `BookingWidget`, and `OTPInputBox` all take submission behavior as an injected
  `onSubmit`/`onVerify` callback prop rather than calling anything themselves — meaning these
  three components require **zero changes** when real API calls are wired in; only the parent
  screen's callback implementation needs to change. This is a well-designed inversion of control
  for a pre-backend build.
- Error-shape contracts are already typed ahead of the real endpoints existing (`BookingSubmitError`,
  `OtpVerifyErrorCode`), which will make wiring real responses a matter of satisfying an existing
  interface rather than inventing one under pressure later.

### Weaknesses
- `MOCK_SLOTS` lives inside `app/booking/[id].tsx` rather than `src/data/`, inconsistent with
  `MOCK_DOCTOR`'s placement — no single convention for "where does mock data live."
- No abstraction exists for what a "backend base URL" or API client configuration would look like
  — confirmed there is no `EXPO_PUBLIC_API_BASE_URL` or equivalent anywhere, consistent with the
  README's audit-phase finding that no environment variables are currently required.
- The three inversion-of-control seams (`onSubmit` etc.) are a good pattern per-component, but
  there is no *shared* typed HTTP-client abstraction they'll eventually all call into — each
  screen's future real implementation could easily diverge in error-handling style without one.

### Risks
- **High** (organizational, not functional): `docs/15-Known-Gaps.md` documents at least 6
  Phase-0 backend gaps (cookie-only auth incompatible with mobile Bearer tokens, missing
  waitlist-join endpoint, no consent-log write path, etc.) that block real integration entirely —
  this is a backend readiness risk, not a mobile-code risk, but it means the mobile "API Layer"
  section of this audit cannot be scored against a working integration, only against
  readiness/seams.
- **Medium**: without a shared HTTP client now, the first real integration pass is likely to
  create 3 slightly different fetch patterns (one per screen) rather than one reusable one.

### Recommendations
- Move `MOCK_SLOTS` into `src/data/` alongside `mockDoctor.ts` for consistency.
- Scaffold a single `src/services/apiClient.ts` (even a thin `fetch` wrapper with base-URL config
  and a consistent error-shape normalizer) now, so the first real integration has one obvious
  place to plug in rather than three independent ad-hoc implementations.
- Keep the existing `onSubmit`-callback inversion-of-control pattern for components — it is
  working well and should be the template for any new form/action component.

## 6. Theme Architecture

### Current State
Theming is implemented as a **static Tailwind config**, not a runtime theme provider. All design
tokens (colors) are hardcoded into `tailwind.config.js`'s `theme.extend.colors` as a flat 1:1 map
of the hex values documented in `docs/08-Design-System.md`. `global.css` contains only the three
standard `@tailwind` directives — no CSS custom properties, no dark-mode variants. Typography and
spacing are **not tokenized** at the config level at all — every component hardcodes literal pixel
values via NativeWind arbitrary-value classes (e.g. `text-[18px]`, `leading-[23px]`,
`rounded-[12px]`) rather than referencing a shared `text-heading` or `spacing-md` token.

### Strengths
- Color tokens are genuinely centralized and well-named (`primary`, `textSecondary`,
  `statusAvailable`, etc.) — no component hardcodes a raw hex color for anything that has a
  semantic token equivalent (spot-checked across all 10 components; the few raw hex values found,
  e.g. `#6B7280` for icon colors in `DoctorCard.tsx`/`QueueStatusBadge.tsx`, are icon `color` props
  which NativeWind's `className` cannot style directly — a real platform limitation, not an
  oversight).
- Every component file carries an inline comment citing the exact `docs/08-Design-System.md`
  section its styling choice maps to — this is unusually strong traceability for a young codebase
  and will make a future design-token refactor much safer than in a typical project.
- `docs/15-Known-Gaps.md` explicitly confirms dark mode is out of scope for mobile by design (web
  admin/doctor dashboards only) — so the *absence* of a dark-mode-capable theme provider is a
  documented decision, not an oversight.

### Weaknesses
- No `ThemeProvider`/`useTheme()` hook exists — colors are consumed only through Tailwind
  class names baked in at author-time. There is no way for a component to read "the current
  primary color" programmatically (needed, e.g., for the `Avatar`/`Badge`/icon `color` props that
  can't take a `className`, currently solved by re-hardcoding hex strings like `'#5696C7'`
  directly in multiple files — `Avatar.tsx`, `Button.tsx`, `DoctorCard.tsx`, `Input.tsx` all
  independently hardcode `#5696C7` for the same "primary" concept instead of importing one shared
  constant).
- Typography scale and spacing scale are **not tokenized at all** — every font size/line-height/
  radius is a NativeWind arbitrary value (`text-[14px]`, `leading-[21px]`) repeated dozens of times
  across files rather than a `text-body`/`text-heading` Tailwind theme extension. This is the
  single largest theme-architecture gap: colors are done well, typography/spacing are not done at
  all.

### Risks
- **High**: `#5696C7` (primary blue) is hardcoded as a raw string in at least 4 different
  component files for icon-color props. If the brand color ever changes, this requires a
  multi-file find-and-replace with no compiler-enforced single source of truth — exactly the
  problem the `tailwind.config.js` color tokens were built to prevent for `className` usage, but
  which doesn't extend to `color` prop usage.
- **Medium**: the lack of a typography/spacing token scale means any future global type-scale
  change (e.g. a font-size accessibility setting, a rebrand) requires touching every component
  file individually rather than one config file.

### Recommendations
- Create a small `src/theme/colors.ts` (or similar) exporting the same hex values already in
  `tailwind.config.js` as named JS constants, and have every `color="#5696C7"`-style icon prop
  import from it instead of re-hardcoding the string. This closes the gap between "tokenized for
  `className`" and "tokenized for `color` props" with minimal risk.
- Extend `tailwind.config.js`'s `theme.extend` with `fontSize` and `borderRadius` scales matching
  the values already repeated everywhere (`14px`/`16px`/`18px`/`24px` for text; `8px`/`12px`/
  `16px`/`20px` for radius), then migrate components incrementally — this is a mechanical,
  low-risk refactor once the scale exists.

## 7. Atomic Design

### Current State
| Layer | Count | Examples |
|---|---|---|
| Atoms | 5 | `Avatar`, `Badge`, `Button`, `Input`, `OTPInput` |
| Molecules | 4 | `BookingWidget`, `DoctorCard`, `OTPInputBox`, `QueueStatusBadge` |
| Organisms | 1 | `WaitlistForm` |
| Templates | 0 | none |
| Screens | 3 | Home, Doctor Profile, Booking |

### Strengths
- Layering discipline is **genuinely correct**, not just folder-deep organization theater:
  - Atoms (`Avatar`, `Badge`, `Button`, `Input`, `OTPInput`) import only React Native primitives,
    Lucide icons, and the shared `AccessibleProps` type — never each other, never a molecule.
  - Molecules (`DoctorCard`, `OTPInputBox`, `QueueStatusBadge`) import only atoms (and, in
    `DoctorCard`'s case, one sibling molecule, `QueueStatusBadge` — an acceptable
    molecule-composes-molecule pattern).
  - The one organism (`WaitlistForm`) imports only atoms.
  - Verified with a full read of every import statement in all 10 component files — zero upward
    or lateral layering violations found.
- A known naming collision (`BookingWidget` molecule vs. a documented `BookingWidget` *organism*
  in `docs/09-Component-Library.md` with a completely different props contract) was caught and
  explicitly flagged in both the component file's own header comment and
  `docs/15-Known-Gaps.md`, rather than silently overwriting the spec or the code. This is a
  strong example of documentation-code consistency discipline that most young codebases lack.

### Weaknesses
- **No Templates layer exists.** All 3 screens compose molecules/organisms directly with raw
  `ScrollView`/`View`/`Text` layout rather than through a reusable screen-template component (e.g.
  a `ScreenScrollContainer` template wrapping the repeated
  `className="flex-1 bg-background"` + `contentContainerClassName="gap-6 px-4 py-6"` pattern,
  which is currently copy-pasted verbatim across all 3 screen files).
- Only 10 components exist against a documented `docs/09-Component-Library.md` that implies a much
  larger eventual set — the library is real but very early; most named components in that doc
  don't have a corresponding file yet (not a defect, just a stage-of-completeness fact worth
  stating plainly for the audit).

### Risks
- **Low–Medium**: the copy-pasted screen-container `className` string (`flex-1 bg-background` /
  `gap-6 px-4 py-6`) across all 3 screens is a small duplication now but will multiply linearly
  with every new screen if a Template layer isn't introduced before screen count grows
  significantly past the documented 16-screen inventory.

### Recommendations
- Introduce `src/components/templates/ScreenContainer.tsx` (or similar) now, while only 3 screens
  need migrating, wrapping the repeated `ScrollView` + `className` pattern — this is the single
  highest-leverage, lowest-risk structural improvement available given the current codebase size.
- No changes needed to the atoms/molecules/organisms layering itself — it is correct and should be
  preserved exactly as-is going forward.

## 8. Shared Utilities

### Current State
**No `src/utils/` folder exists.** Utility-style pure functions exist but are private to
individual component files:
- `formatFee`, `initialsFromName` — private to `DoctorCard.tsx`
- `maskPhone` — private to `OTPInputBox.tsx`
- `digitsOnly`, `isNameValid`, `isPhoneValid` — private to `WaitlistForm.tsx`
- `valueToCells`, `cellsToValue` — private to `OTPInput.tsx`
- `formatErrorCodeLabel` — private to `app/booking/[id].tsx`

### Strengths
- Every function found is genuinely pure (no hidden side effects, no closures over component
  state) — they are trivially extractable to a shared module with no refactor risk beyond import
  paths.
- Naming is consistent and self-explanatory across files (`isXValid`, `formatX`, `mapX`) — a good
  sign for future consolidation.

### Weaknesses
- Phone-number handling is implemented **three separate times** with three different levels of
  correctness: `WaitlistForm.tsx`'s `digitsOnly`/`isPhoneValid` (strict 10-digit check),
  `OTPInputBox.tsx`'s `maskPhone` (takes last 10 digits, no validation), and no phone logic at all
  in `app/index.tsx` despite it also collecting a phone number for the waitlist form. Any future
  bug fix (e.g. supporting a `+91` prefix already in the string) needs to be applied in multiple
  places to stay consistent, with no compiler or test to catch a missed spot.

### Risks
- **Medium**: phone-validation duplication is the clearest concrete instance of the general "no
  shared utilities" risk — it already has 2 independent implementations at only 3 screens deep.

### Recommendations
- Create `src/utils/phone.ts` exporting `digitsOnly`, `isPhoneValid`, and `maskPhone` as the one
  shared implementation; update `WaitlistForm.tsx` and `OTPInputBox.tsx` to import from it.
- Create `src/utils/format.ts` for `formatFee`/`initialsFromName`, anticipating that a future
  `DoctorList`/`DoctorSearch` screen will need the same formatting `DoctorCard` already has.

## 9. Hooks

### Current State
**No custom hooks exist anywhere in the codebase.** No `src/hooks/` folder, no `use*.ts` files
outside of React Native's/React's own built-ins (`useState`, `useEffect`, `useRef`, `useCallback`
— all used directly inline in components).

### Strengths
- Where multi-step stateful logic exists (`OTPInput`'s digit-cell management, `OTPInputBox`'s
  resend-countdown announcement logic), it is at least isolated to the one component that owns it
  — there's no cross-component copy-paste of hook-like logic to point to as a defect, because
  nothing has been duplicated (yet) at the hook level.

### Weaknesses
- `OTPInput.tsx`'s internal digit-cell state machine (~70 lines: `valueToCells`, `cellsToValue`,
  `handleChange`, `handleKeyPress`, ref-focus management) is exactly the shape of logic that
  usually becomes a `useOtpCells()` custom hook once a second OTP-style component needs the same
  behavior — currently fine as inline logic since there's only one consumer, but worth watching.
- The pattern of "fetch → loading/error/success state machine" appears independently in both
  `BookingWidget.handleBookPress` and the Home screen's `handleWaitlistSubmit` — a natural
  `useAsyncAction()`-style hook candidate once a third instance appears.

### Risks
- **Low** (currently): with zero duplication today, this is a "watch this" item, not an active
  problem.

### Recommendations
- No action needed yet. Revisit once a second OTP-entry component or a third async-submission
  flow is added — extracting a hook a mechanically-relevant amount too early would add
  abstraction the codebase doesn't need yet.

## 10. Services

### Current State
**No services layer exists.** No `src/services/` folder, no network client, no analytics client,
no notification service, no storage/persistence wrapper (no AsyncStorage/SecureStore usage found
anywhere despite these being common Expo patterns).

### Strengths
- N/A — nothing exists to evaluate positively beyond "nothing incorrect exists either," since
  there is no premature/wrong abstraction to critique.

### Weaknesses
- Given `docs/11-API-Contract.md`'s extensive endpoint documentation and `docs/15-Known-Gaps.md`'s
  list of pending integrations, the complete absence of even a placeholder services folder means
  there is no established convention for where the first real service module should go.

### Risks
- **Medium**: without an established `src/services/` convention, the first developer to wire a
  real endpoint has no precedent to follow and may reasonably choose ad-hoc, inconsistent
  placement (e.g. directly inside a screen file, as the current mock submissions already do).

### Recommendations
- See Section 5 (API Layer) — the same `src/services/apiClient.ts` recommendation applies here.
  Additionally scaffold `src/services/storage.ts` as a thin AsyncStorage/SecureStore wrapper
  before any auth-token persistence is needed, so token storage doesn't get implemented ad-hoc
  inside a login screen later.

## 11. Constants

### Current State
**No `src/constants/` folder exists.** The closest things to constants are:
- `BOOKING_ERROR_CODES` (`BookingWidget.tsx`) — a well-formed exported constant, but scoped to one
  component file rather than a shared constants module.
- `MOCK_SUBMIT_DELAY_MS`, `PHONE_DIGIT_COUNT`, `MIN_NAME_LENGTH`, `DEFAULT_LENGTH` — scattered,
  module-local constants in various files, each single-use today.
- Route path strings (`/doctor/${id}`, `/booking/${id}`) are hand-typed template literals at every
  `router.push` call site rather than a typed route-constants module.

### Strengths
- Constants that do exist are correctly typed and exported where reuse is anticipated
  (`BOOKING_ERROR_CODES` is explicitly exported "for exhaustiveness checks / tests").

### Weaknesses
- Route strings are the clearest gap: `/doctor/${doctor.id}` is typed out independently in
  `app/index.tsx` and `app/doctor/[id].tsx`; `/booking/${doctor.id}` similarly appears in both
  `app/index.tsx` and `app/doctor/[id].tsx`. A typo in one call site (e.g. `/doctors/` plural)
  would not be caught by TypeScript, since Expo Router's typed-routes feature is not enabled in
  `app.json`/`tsconfig.json` (confirmed — no `experiments.typedRoutes` entry exists in `app.json`).

### Risks
- **Medium**: route-string duplication without typed-routes support is a silent-failure risk —
  a renamed dynamic segment folder (e.g. `doctor/[id]` → `doctors/[doctorId]`) would not produce
  any compile error at the two call sites still using the old string shape.

### Recommendations
- Enable Expo Router's `experiments.typedRoutes` in `app.json` (a config-only change, no source
  restructuring needed) to get compile-time route-string checking for free.
- Create `src/constants/routes.ts` with named route-builder functions (`doctorRoute(id)`,
  `bookingRoute(id)`) even before typed routes are enabled, as a lower-effort intermediate step.

## 12. Types

### Current State
One shared types file exists: `src/types/accessibility.ts`, exporting `AccessibleProps` and
`AccessibilityRoleToken`, consumed by every atom/molecule/organism. All other types (`AvatarSize`,
`ButtonVariant`, `DoctorCardDoctor`, `BookingErrorCode`, etc.) are defined and exported directly
alongside the component that owns them, via each folder's `index.ts` barrel re-export.

### Strengths
- The single shared type (`AccessibleProps`) is genuinely well-designed: a single interface
  capturing the accessibility contract every interactive component must satisfy, extended by every
  component's own props interface. This is exactly the right amount of centralization — one true
  cross-cutting concern (accessibility), centralized; everything else, co-located with its owner.
- TypeScript strict mode is enabled (`tsconfig.json`: `"strict": true`) and the full project
  compiles with **zero errors** (verified in Phase 1 of this audit via `npx tsc --noEmit`).
- Barrel exports (`index.ts` per folder) consistently re-export both the component and its types
  together (e.g. `export { DoctorCard, type DoctorCardProps, type DoctorCardDoctor } from
  './DoctorCard'`), so consumers never need to reach into a component's internal file path to get
  a type.

### Weaknesses
- `DoctorCardDoctor` (the doctor domain shape) is defined inside `DoctorCard.tsx`, a UI component
  file, rather than in a domain-level `src/types/doctor.ts`. This means the "doctor" domain
  entity's shape is owned by a presentation component — a minor Clean-Architecture violation
  (Section 2) surfacing again at the type level: the domain model should not need a UI component
  to exist in order to be typed.
- No shared `src/types/api.ts` or similar exists for future API response envelopes, despite
  `docs/11-API-Contract.md` documenting two different response envelope formats already
  (`apiResponse`-wrapped vs. flat) that mobile code will eventually need to type against.

### Risks
- **Low**: current type organization works fine at this scale; the domain-type-in-UI-file issue
  is a "will hurt later" risk, not a current one, since there's only one doctor-shaped screen data
  source right now.

### Recommendations
- Move `DoctorCardDoctor` to `src/types/doctor.ts` (or a broader `src/types/domain.ts`) and have
  `DoctorCard.tsx` import it, freeing the domain shape from a UI component's file so a future
  non-card doctor-list/search screen doesn't need to import a "card" component just to get the
  doctor type.

## 13. Assets

### Current State
`assets/brand/` contains 24 files (a mix of `.svg` source files and their exported `.png`/`.ico`
derivatives) organized into `app-icon/`, `brand-icon/`, `examples/`, `favicon/`, `monochrome/`,
`primary-logo/`, `wordmark/` subfolders. No UI-specific imagery (illustrations, empty-state
graphics, onboarding art) exists — the only assets are brand identity files.

### Strengths
- Assets are organized by *purpose* (app-icon vs. favicon vs. wordmark), not dumped flat — a
  sensible structure that will scale reasonably as more brand variants are added.
- `docs/08.1-Brand-Assets.md` is the single documented source of truth for these assets per the
  workspace's own `.cursorrules`, avoiding the common problem of asset usage rules being scattered
  or undocumented.

### Weaknesses
- `app.json`'s Expo config does **not reference any of these assets** — no `icon`, `splash`, or
  `adaptiveIcon.foregroundImage` fields point at anything in `assets/brand/`. The Android
  `adaptiveIcon` config only sets a `backgroundColor`, with no `foregroundImage`. This means the
  app currently builds with Expo's default icon/splash, not the JivniCare brand assets that
  already exist in the repo — the brand assets and the app configuration are currently
  disconnected.
- Several filenames contain spaces (`"primary logo black.svg"`, `"primary-logo -vertical.svg"`,
  `"JivniCare wordmark.png"`) — not a build-breaking issue (already verified: the app bundles
  successfully in Phase 1 of this audit), but inconsistent with the hyphenated-kebab-case
  convention used everywhere else in the repo (`app-icon-1024.svg`, `brand-icon-black.svg`) and a
  minor source of friction for anything that shells out to a filename (scripts, CLI asset
  pipelines).
- Both a `.svg` and pre-rendered `.png` exist for most assets — no build step was found that
  generates one from the other, so these are two independently-committed copies that could drift
  out of sync if one is updated without the other.

### Risks
- **Medium**: the app.json/assets disconnect means the app is currently shipping with a generic
  Expo icon/splash screen despite brand assets existing — this is likely to surprise whoever next
  runs a build and expects to see the JivniCare logo.
- **Low**: filename-space inconsistency is cosmetic but will eventually cause friction in a CI
  script or asset-optimization pipeline that doesn't robustly quote paths.

### Recommendations
- Wire `app.json`'s `icon`, `splash`, and `android.adaptiveIcon.foregroundImage` fields to the
  appropriate files already present in `assets/brand/app-icon/`.
- Standardize on either an all-`.svg`-source-with-generated-`.png` pipeline, or commit to
  maintaining both manually with a documented process — currently neither is explicit.
- Rename space-containing filenames to kebab-case for consistency (a mechanical, low-risk rename;
  would need corresponding reference updates if/when `app.json` is wired to point at them).

## 14. Performance Architecture

### Current State
No explicit performance-oriented architecture exists yet — no `React.memo`, no `useMemo`/
`useCallback` used for render-cost optimization (the few `useCallback` usages in `OTPInput.tsx`
are for stable prop-identity to child refs, not render-cost optimization per se), no list
virtualization (no `FlatList`/`FlashList` usage anywhere — unsurprising, since no screen renders
a list yet; `app/index.tsx` renders exactly one `DoctorCard`), no image-loading optimization
(`Avatar.tsx`'s `<Image>` usage has no `expo-image` caching layer — it uses React Native's base
`Image` component).

### Strengths
- `react-native-reanimated`/`react-native-worklets` are already installed dependencies, meaning
  the project has correctly anticipated needing native-thread animation performance ahead of
  actually building animation-heavy screens — a reasonable early investment. (Currently unused:
  `OTPInput.tsx`'s shake animation uses React Native's core `Animated` API, not Reanimated —
  consistent, not a defect, since Reanimated isn't needed for a simple shake.)
- `Avatar`'s `<Image>` usage correctly sets `resizeMode="cover"` and constrains dimensions via a
  fixed-size container — basic image-performance hygiene is present even without a caching layer.

### Weaknesses
- No `FlatList`/virtualized-list strategy is established, which will matter immediately once a
  Search/Doctor-listing screen (documented in `docs/07-Mobile-UX-Spec.md`'s screen inventory) is
  built — `docs/15-Known-Gaps.md` Section 4 notes a 100-doctor search cap on the *backend* side,
  but there is no corresponding mobile-side plan for rendering that list performantly.
- Base `Image` (not `expo-image`) means no memory-caching, no placeholder/blurhash support, no
  automatic format negotiation — this will matter once `Avatar`'s `source` prop is fed real doctor
  photo URLs instead of always being `undefined` (as it is today in `mockDoctor.ts`).

### Risks
- **Low** (currently — no lists, no real images yet): this section is entirely forward-looking.
  The risk is retrofitting virtualization/image-caching onto an already-built Search screen later,
  rather than any current defect.

### Recommendations
- Adopt `expo-image` (already an Expo-ecosystem-standard, drop-in-compatible-ish replacement) for
  `Avatar.tsx`'s `<Image>` before real photo URLs are wired in.
- Plan for `FlatList` (or `@shopify/flash-list`) explicitly when the first list-rendering screen
  (Search/Doctor listing) is built — not a current-code issue, but worth flagging now given the
  backend's own 100-item cap implies non-trivial list sizes are expected.
